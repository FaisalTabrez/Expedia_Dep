# M2.5 SDK and REST adapter conformance

**Status:** Implemented local M2.5 adapters.  
**Governing specification:** EDS v2.1.1 §11–12; ADR-010 and ADR-016.  
**Boundary:** SDK and REST receive an injected Query Core; they never open a
release package or create a semantic result independently.

## Python SDK

`LocalExpediaClient` is the typed in-process wrapper. It serializes a supplied
mapping to JSON and invokes `QueryCore.execute`. The returned mapping is the
Core QueryResult. It contains no release location, profile resolution, filter,
ranking, cursor, provenance, warning, or error implementation.

## REST transport

`V1QueryApplication` is a dependency-free local WSGI application with one
route: `POST /v1/query`. It forwards UTF-8 request JSON to Core and serializes
the returned envelope under HTTP 200, including typed query-error envelopes.
HTTP 400/404/405 are restricted to malformed transport input or route/method
errors before a QueryRequest can reach Core. The associated OpenAPI contract is
`api/openapi/v1/openapi.yaml`.

There is no remote server deployment, authentication, authorization, rate
limit, HTTP client, or service availability claim in M2.5.

## Conformance evidence

`tests/conformance/test_m2_5_adapter_conformance.py` builds a verified corrected
M1 successor fixture and executes the identical corpus through Core, SDK, and
REST. It compares successful filtered queries, typed unsupported-filter errors,
and cursor continuation envelopes byte-for-logical-content equality. It also
asserts the adapter sources do not open releases or implement Core behavior.
