# M2.2 Verified Release Adapter

**Status:** Implemented for the local M1 Draft conformance fixture.
**Governing sources:** EDS v2.1.1 sections 11, 12.1, and 15.3; ADR-010.

## Purpose

`open_verified_release()` is the sole M2.2 entry point from a local package
directory into Query Core. It either returns one immutable `VerifiedRelease`
snapshot or raises `ReleaseVerificationFailure`. It is intentionally a small,
auditable trust boundary, not a query engine.

## Trusted-open procedure

1. Reject a missing or symbolic-link package root.
2. Invoke the existing M1 Release Reader. That reader validates the Draft
   ReleaseManifest, embedded schema compatibility, manifest inventory, artifact
   digests and sizes, safe paths, M1 provenance policy, and vector integrity.
3. Re-read the root manifest and require its SHA-256 to equal the reader's
   verified manifest digest.
4. Re-check and snapshot every manifest-addressed payload byte. Symbolic links,
   escapes from the package root, missing artifacts, size mismatches, and digest
   mismatches fail closed.
5. Expose only immutable metadata, JSONL-table rows, and profile-scoped vector
   payloads from that snapshot. The handle never reads the package directory
   again.

## Failure contract

`ReleaseVerificationFailure.code` is one of the accepted M2 QueryError values:

| Code | Meaning |
|---|---|
| `release_not_found` | The requested local package directory does not exist. |
| `release_untrusted` | Verification, schema compatibility, path safety, or snapshot integrity failed. |

A failed open returns no records, vectors, or trusted release handle. It does
not turn the M1 Draft fixture into a public, citable, or Published release.

## Deliberate non-goals

M2.2 contains no query execution, exact cosine score calculation, vector
decoding, filtering, traversal, cursor behavior, cache, SDK, REST, Explorer,
ANN/index adapter, network access, or release mutation. Those concerns remain
in their assigned later milestones.
