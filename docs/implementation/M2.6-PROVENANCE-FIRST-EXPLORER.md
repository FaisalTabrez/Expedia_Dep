# M2.6 provenance-first Explorer

**Status:** Implemented local M2.6 presentation boundary.  
**Governing specification:** EDS v2.1.1 §10 and §12; ADR-016.  
**Boundary:** Explorer receives a QueryResult source and presents the returned
envelope; it does not read or query a release independently.

## Presentation model

`ProvenanceExplorer` accepts an injected result-source callable, normally
`LocalExpediaClient.query`. `query_and_present` forwards the request unchanged
to that source and renders the returned QueryResult. `present_result` may also
render a supplied Core result directly.

Successful views show release ID/digest and integrity evidence, selected
profile, metric/direction, mode, ordering policy, request digest, complete
provenance, evidence status, warnings, cursor, and rows. Similarity rows are
explicitly labeled `canonical-record` and preserve their Core-provided order.
Typed QueryResult errors remain typed presentation errors; they are not
translated into alternate Explorer semantics.

Scope, citation, and compatibility are displayed only when a trusted caller
supplies them to `present_release`. QueryResult alone does not supply these
items, so the view labels them `not-provided-by-query-result` rather than
inventing values. The M1 Draft QueryResult also has no annotation assertions,
relation lineage, or projection payload; Explorer states those absences
explicitly.

## Exclusions

M2.6 does not prescribe a desktop or web framework, process isolation, remote
service client, release reader, filter engine, ranking engine, cursor logic,
annotation workflow, graph view, projection, or biological interpretation.

## Conformance evidence

`tests/conformance/test_m2_6_explorer_conformance.py` runs the Explorer over a
verified corrected M1 successor fixture through the local SDK. It verifies that
presentation preserves Core result rows/context/evidence, preserves a typed
unsupported-filter error, displays only caller-supplied release context, and
contains no release/query semantic implementation.
